# `frontend/dist/` chunk-size baseline

This is the committed bundle-size reference for the map-first re-architecture
(epic #761). Later workstreams — WS8 (prefetch maplibre/MapCanvas on scoped
paths, sub-issue 8.1), the eager-vs-lazy bundle decision (OQ5), and the epic's
"`dist/` chunk sizes are recorded with the `maplibre-gl` cost called out"
acceptance criterion — measure cold-start regressions against the numbers
recorded here.

**Intent (one line):** this is the reference the map-first bundle-strategy work
(OQ5 / WS8) measures cold-start regressions against; eager-inlining the
`maplibre-gl` chunk into the entry bundle would add ~273 kB gzip (~1,028 kB raw)
to **every** cold start, including the unscoped chooser landing that never
touches the map today.

## How this baseline was produced

| Field | Value |
|---|---|
| `git rev-parse HEAD` | `0fc7c64028d96105497875201c71cb257bce623c` |
| `node --version` | `v24.4.1` |
| `npm --version` | `11.4.2` |

These sizes come from an **actual reproduced local build**, not transcribed
numbers. `frontend/dist/` is **gitignored** (`git check-ignore frontend/dist`
matches; `git ls-files frontend/dist` returns nothing), so the bytes below are
not in the committed tree — they are regenerated by running the reproduce
command. Content hashes in the emitted filenames (e.g. `index-B1RWV4Bm.js`,
`maplibre-gl-CxlP_mE2.js`) are **build-dependent** and drift on any clean
rebuild; this doc therefore cites **chunk sizes**, not hash-pinned filenames,
as the stable reference.

## Reproduce

```sh
# from repo root — shared-types must be built first so Rolldown can resolve it
npm run build --workspace @bird-watch/shared-types
npm run build --workspace @bird-watch/frontend

# raw bytes
ls -la frontend/dist/assets/index-*.js frontend/dist/assets/maplibre-gl-*.js

# gzip bytes (per file)
gzip -c frontend/dist/assets/index-*.js | wc -c
gzip -c frontend/dist/assets/maplibre-gl-*.js | wc -c
```

## Chunk sizes (baseline)

| Chunk | Raw | Gzip |
|---|---|---|
| `index-<hash>.js` (entry) | 208,974 B (~209 kB) | 65,760 B (~66 kB) |
| `maplibre-gl-<hash>.js` (lazy) | 1,027,573 B (~1,028 kB) | 270,095 B (~273 kB) |

**`maplibre-gl` cost called out:** the heavy mapping dependency is the
`maplibre-gl-<hash>.js` chunk at **~1,028 kB raw / ~273 kB gzip** — roughly
4.9× the entry chunk's raw size. It is loaded **lazily**: the `React.lazy()`
boundary in `frontend/src/components/MapSurface.tsx` fetches it only when
`MapCanvas` first renders, so the unscoped chooser landing never pays for it.
This is the figure every downstream bundle decision keys on (OQ5 / R4) — the
in-code comment previously understated it as "217 KB", which this baseline and
the corrected comment replace.

## Entry-chunk maplibre-ref measurement (default split is clean)

The default Vite/Rolldown code split already keeps `maplibre-gl` out of the
entry chunk. This is recorded as a **reproducible measurement** (command +
observed result), not a hardcoded constant — a later intentional split change
(WS8 / sub-issue 8.1 scope-gated prefetch, or an added `manualChunks`)
re-baselines by re-running the command rather than silently falsifying a
literal:

```sh
grep -c maplibre frontend/dist/assets/index-*.js
# observed: 0
```

`frontend/vite.config.ts` has **no** `manualChunks` / `rollupOptions` /
`rolldownOptions` — the split above is the default Vite/Rolldown behavior, not a
hand-tuned configuration. (The config's only `resolve.alias` entry pins
`maplibre-gl` to the frontend's local copy per issue #258; it does not affect
chunking.)
