1. **react-map-gl v7 `interactiveLayerIds` + `onClick` does not populate `e.features`** when layers are added via JSX `<Source>`/`<Layer>` children. The fix is to call `map.getMap().queryRenderedFeatures(point, { layers })` inside the `onClick` handler using the raw MapLibre instance. Plan 7's `MapCanvas.tsx` must use this pattern, not the `e.features` approach shown in most react-map-gl examples.
2. **OpenFreeMap's `liberty` style has an upstream bug** that emits `"Expected value to be of type number, but found null"` warnings from MapLibre internals. `positron` is clean at initial zoom but also warns at zoom >7. The production PMTiles basemap must validate its style spec to avoid inheriting this — add a CI check or use a MapLibre style linter.
3. **MapLibre GL JS ships at 217 KB gz** (v4.7, standalone). Combined with react-map-gl (~65 KB gz for the full React bundle including our code), the total map chunk is ~282 KB gz. Under the 250 KB threshold for MapLibre alone. Lazy-loading via `React.lazy()` is essential to keep Feed/Species tab LCP unaffected.
4. **Mobile 390x844 renders well** — clusters are large enough to tap, and the map fills the viewport cleanly. The attribution bar steals ~40px at the bottom; no layout issues. Individual unclustered points at 7px radius are too small for reliable touch targets — production should bump to 10-12px or add a tolerance ring on mobile.
5. **Would do differently:** skip react-map-gl's JSX `onClick`/`interactiveLayerIds` entirely and wire all interaction through `onLoad` → `map.on('click', layerId, handler)`. The JSX abstraction adds indirection without value for click handling and required debugging a feature-population bug that cost ~1 hour. The JSX `<Source>`/`<Layer>` declarative pattern is still valuable for rendering.
