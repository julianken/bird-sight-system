# Migrations

Plain SQL migrations run by `node-pg-migrate`. Each file uses
`-- Up Migration` and `-- Down Migration` markers.

## Naming convention

Files follow the pattern `1700000NNN000_<slug>.sql` where `NNN` is a
zero-padded three-digit sequence number (001–999). The leading `1700000`
prefix and trailing `000` suffix are fixed — do not change them. The sequence
must be strictly ascending; gaps are acceptable (see below).

```
1700000001000_enable_postgis.sql      ← first migration
1700000002000_regions.sql
...
1700000019000_add_common_name_to_family_silhouettes.sql
```

## The `1700000010000` gap

The sequence jumps from `1700000009000` to `1700000011000`. The `10000` slot
was never reserved and must not be used. Do not fill this gap; leave it empty
to avoid renumbering the migrations that follow.

## The `.x500` / `.x700` interpolated suffix convention

Out-of-band migrations that need to be inserted between two already-committed
natural-sequence slots use an interpolated suffix: `.x500` or `.x700` added
between the base number and the trailing `000`.

Examples already in the sequence:

| File | Inserted between |
|---|---|
| `1700000019500_seed_family_common_names.sql` | `19000` and the next natural `20000` |
| `1700000019700_seed_fallback_common_name.sql` | `19000` and the next natural `20000` (after `19500`) |

When choosing a suffix for a new out-of-band migration:

- Use `.x500` for the first insertion in a gap, `.x700` for the second.
- Do not create a third insertion in the same gap — reserve a clean natural
  slot instead.
- The naming must produce a file whose lexicographic sort order places it
  correctly between its neighbors.

## Schema-evolution reading rule

DDL in each migration reflects the correct state at the time that migration was
written. Later migrations may alter that state. For example:

- `1700000003000_family_silhouettes.sql` creates `family_silhouettes` with
  `svg_data NOT NULL`.
- `1700000014000_relax_family_silhouettes_svg_data_nullable.sql` drops that
  constraint.

The intermediate DDL in `03000` is not wrong — it was correct when it was
committed. But the operative constraint today is the one established by the
last migration that touched the column. To understand the live schema, read
the full migration sequence, not just the create-table migration.

## Down migrations

Every migration must include a reversible `-- Down Migration` block. Reversibility is
enforced by `packages/db-client/src/migrations-down-chain.test.ts`, which applies all migrations
up and then reverses them one-by-one, verifying each step succeeds.

Per issue #301: down migrations that were previously no-ops were corrected
to be truly reversible. Do not write placeholder down migrations.
